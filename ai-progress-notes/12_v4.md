## Baseline Before Fixes

The original baseline was:

- `6 PASS`
- `3 FAIL`

The three failures were:

1. `0xf265f5c9` ‚Äî V4 route reverted
2. `0x21a71a54` ‚Äî same class of V4 revert
3. `0x1727607d` ‚Äî `-4.54%` output gap

At that point it looked like the LFJ V4 failures were caused by matching a zero-liquidity `BTC.b/USDt` V4 pool. That turned out to be an incorrect diagnosis caused by bad route reconstruction.

## Root Cause 1: V4 Pool Matching By Transfer Flow Was Wrong

The first real bug was in `examples/03_route_analyzer/convert.ts`.

We were reconstructing V4 steps from ERC-20 `Transfer` logs involving the PoolManager singleton:

- PoolManager receives input during `sync()` / `settle()`
- PoolManager sends output during `take()`
- transfer patterns alone do not uniquely identify the V4 pool
- native-token pools make this even worse because the native side does not emit an ERC-20 transfer

That produced fake routes like:

- inferred: `BTC.b -> USDt`
- actual: `BTC.b -> AVAX` in a native V4 pool

### What The Trace Actually Showed

For the LFJ failures:

- tx `0xf265f5c9` emitted PoolManager `Swap` log with `topic1 = 0xa685e30445ba26e6b744e45f636fccab0d44969c873a85564cfaf4be5a61eb14`
- that poolId maps to V4 pool `AVAX/BTC.b`

For the Odos V4 case:

- tx `0x1727607d` emitted PoolManager `Swap` log with `topic1 = 0x6874d3d2b99e6b75ed2fae390907e1d921d4b06b7f48fc6d3fe9c547507e369c`
- that poolId maps to V4 pool `AVAX/USDC`

So the original "wrong BTC.b/USDt zero-liquidity V4 pool" was not the real route. It was an artifact of reconstructing PoolManager activity from transfers instead of from V4 swap logs.

## Fix 1: Reconstruct V4 From PoolManager Swap Logs

Implemented in `examples/03_route_analyzer/convert.ts`:

- added V4 `Swap` topic parsing
- extract actual `poolId` from PoolManager `Swap` logs in the replay trace
- map `poolId` to V4 metadata from `pools.txt`
- build V4 steps directly from that pool metadata

This is the correct source of truth. It is stronger than token-pair heuristics.

### Defensive Fallback Hardening

We also hardened the old V4 fallback path in `findPool()` so it is less likely to pick garbage if it is ever used:

- parse `id`, `fee`, and `hooks` from V4 extra data
- reject hooked V4 pools in fallback mode
- reject absurdly high-fee V4 pools in fallback mode
- among remaining candidates, prefer highest `latestSwapBlock`

This is not the primary fix. It is only guardrail logic.

## Root Cause 2: Router Bytecode On-Chain Did Not Include V4 Support

The deployed router contract at:

- `0x2bef1becdafcfe8990a233d03a98bbb39021c96e`

was deployed before V4 callback support was added.

Missing on deployed bytecode:

- `unlockCallback(bytes)` selector `0x91dd7346`
- `v4UnlockCallback()` selector `0x1c4695f4`

Working workaround:

- `router/quote.ts` injects compiled `router/contracts/bytecode.hex` as a code override on every quote

Without the override, V4 quotes revert almost immediately because the callback selector does not exist on-chain.

## Root Cause 3: Native-Token V4 Legs Needed Real Router Support

Once V4 pool matching was fixed, the LFJ failures stopped being bogus `BTC.b -> USDt` routes and became what they actually were:

- `BTC.b -> AVAX` via V4 `AVAX/BTC.b`

Likewise, the Odos route contained:

- `AVAX -> USDC` via V4 `AVAX/USDC`

At that point skipping was not acceptable. The router needed to support native legs.

## Fix 2: Add Native-Token V4 Support To The Router

Implemented in:

- `router/contracts/HayabusaRouter.sol`
- `router/quote.ts`

### Solidity changes

- `swap()` is now `payable`
- `quoteRoute()` is now `payable`
- if the first token is `address(0)`, input is taken from `msg.value`
- if the final token is `address(0)`, output is returned with an ETH transfer
- V4 balance accounting uses `address(this).balance` for native output
- V4 settlement uses `PoolManager.settle{value: amountIn}()` when input token is native

### Quote path changes

- `eth_call` includes `value` for native-input routes
- dummy sender gets a native balance override for native-input quotes
- code override is still injected for the router itself
- same native-aware setup is used for `eth_estimateGas` and `debug_traceCall`

This is why the previously skipped native V4 routes now execute.

## Root Cause 4: One Missing WooPP Branch

After native V4 support landed:

That branch accounted for the missing output.

## V4 Swap Flow In The Router

Current V4 flow:

```text
Router.swap / Router.quoteRoute
  -> _swapLeg(...)
    -> _swapUniV4(tokenIn, tokenOut, amountIn, extraData)
      -> store _v4TokenIn/_v4TokenOut/_v4AmountIn/_v4ExtraData
      -> measure tokenOut balance (ERC-20 or native)
      -> PoolManager.unlock(...)
        -> PoolManager calls Router.unlockCallback(bytes)
          -> Router delegatecalls Router.v4UnlockCallback()
            -> decode fee / tickSpacing / hooks from extraData
            -> build PoolKey
            -> PoolManager.swap(...)
            -> settle input:
               - ERC-20: sync(tokenIn) + transfer + settle()
               - native: sync(address(0)) + settle{value: amountIn}()
            -> take output
      -> return output balance delta
```

Key points:

- V4 uses a callback pattern through PoolManager
- the router delegatecalls itself so inner `msg.sender` remains PoolManager
- native handling is now part of the V4 quote path
